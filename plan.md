
# 全方位学习辅助平台 —— 项目企划书

**项目代号**：OmniLearn (暂定)
**开发者**：独立开发者
**技术栈**：React Native (Expo) + Next.js + Supabase
**核心理念**：工具为先，社区驱动，数据可视化。

---

## 1. 项目愿景与定位 (Vision & Scope)

*   **产品定位**：一款集“时间管理、语言学习、数据激励、同伴交流”于一体的综合性学习伴侣。
*   **目标用户**：高中生、大学生、考研党及终身学习者。
*   **核心价值**：通过量化学习投入（排行榜）和工具辅助（番茄钟/单词），解决学习中的“孤独感”和“反馈滞后”问题。

---

## 2. 软件生命周期规划 (Phased Development)

根据**MVP（最小可行性产品）**原则，将开发分为三个阶段，避免一开始摊子铺太大导致项目流产。

### 第一阶段：单机工具与数据闭环 (MVP)
*   **目标**：解决“能用”的问题，确保核心功能在无网络或弱服务器环境下完美运行。
*   **功能**：
    1.  **专注番茄钟**：自定义时长、白噪音背景、锁屏保活。
    2.  **本地数据统计**：今日学习时长、专注次数分布图。
    3.  **基础背单词**：支持导入本地单词书（JSON/CSV），简单的“认识/不认识”卡片模式。
    4.  **账户系统**：仅做基础的注册/登录（Supabase Auth），为云同步做准备。

### 第二阶段：数据上云与排行榜 (Gamification)
*   **目标**：引入“社交激励”，增加用户粘性。
*   **功能**：
    1.  **数据同步**：将本地的学习记录同步至云端数据库。
    2.  **全服排行榜**：
        *   今日/本周“专注时长”榜单。
        *   今日/本周“听力时长”榜单。
    3.  **听力模块 (基础版)**：仅支持播放本地音频文件或外部链接（如RSS播客），**不自建音频存储**（节省服务器流量）。

### 第三阶段：社区与扩展 (Community & Scale)
*   **目标**：打造生态，尝试高负载功能。
*   **功能**：
    1.  **公共讨论区**：先做“文字版”树洞或打卡区，限制图片上传（降低存储成本）。
    2.  **模块化商店**：允许用户在设置中开启/关闭非核心功能（如：课程表模块、倒数日模块）。

## 3. 功能模块详细设计 (Functional Specs)

### 3.1 专注自习番茄钟 (Core)
*   **设计原则**：极简、防打扰。
*   **细节**：
    *   UI 显示巨大的倒计时。
    *   **技术难点**：后台保活。使用 Expo 的 `KeepAwake` 组件和后台任务 API。
    *   **数据结构**：
        ```json
        { "startTime": "...", "duration": 25, "tag": "Math", "completed": true }
        ```

### 3.2 排行榜系统 (Gamification)
*   **设计原则**：实时性不必太高，减少数据库压力。
*   **逻辑**：
    *   前端每完成一次专注，上传一条 Log 到 `study_logs` 表。
    *   **后端策略**：不要实时 `count` 所有数据。利用 Supabase 的 **Database Views (视图)** 或 **Edge Functions (定时任务)**，每10分钟聚合一次排名前100的数据缓存起来，前端只拉取这个缓存结果。

### 3.3 背单词模块 (Utility)
*   **设计原则**：轻量化，不自建庞大的词库。
*   **策略**：
    *   **开放性**：允许用户导入单词书（标准格式）。App 内置几本基础词书（如四六级）。
    *   **算法**：采用简化的艾宾浩斯遗忘曲线算法（基于本地数据库计算，不需要联网）。

### 3.4 听力模块 (Media)
*   **服务器避坑指南**：音频文件极大，且消耗带宽。
*   **策略**：
    *   **本地优先**：做一个好用的播放器，让用户播放手机里的 MP3。
    *   **流媒体链接**：支持输入在线 MP3 链接或 RSS 订阅源进行播放。
    *   **计时逻辑**：只有当播放器处于“播放状态”时，才计入“听力时长”并参与排行。

### 3.5 公共讨论区 (Community)
*   **策略**：暂缓开发，或极简开发。
*   **形式**：类似“贴吧”或“Twitter”的线性流。
*   **风控**：作为个人开发者，内容审核是最大风险。建议仅开放“固定话题打卡”（如：#今日目标#），限制自由发帖，或者引入举报机制。

## 4. 技术架构与数据库设计 (Architecture)

### 4.1 技术选型
*   **Client (App)**: React Native (Expo) - 跨平台，开发快。
*   **Client (Web)**: Next.js - 只有 PC 端看报表、看帖子的功能，复用 React 组件。
*   **Backend (BaaS)**: **Supabase** (强烈推荐)。
    *   PostgreSQL 数据库：处理排行榜这种复杂查询极快。
    *   Auth：自带用户登录体系。
    *   Realtime：自带即时通讯能力（为了未来的讨论区）。

### 4.2 核心数据库表设计 (ER Diagram 简述)

1.  **`users`**: 用户基本信息 (id, nickname, avatar_url)。
2.  **`study_logs`**: 核心日志表。
    *   `id`, `user_id`, `start_time`, `duration_seconds`, `category` (focus/listening), `tags`.
3.  **`vocab_progress`**: 单词进度表。
    *   `id`, `user_id`, `word_id`, `next_review_time` (下次复习时间), `proficiency` (熟练度)。
4.  **`posts` (后期)**: 帖子表。

---

## 5. UI/UX 设计原则

*   **风格**：**Bento Grid (便当盒风格)**。
    *   首页由几个方块组成：左上角是番茄钟入口，右上角是今日数据概览，下方是排行榜缩略图。
    *   这种设计非常适合“若干可添加模块”的扩展需求，加功能就是加个格子。
*   **交互**：减少层级。核心功能（开始学习）必须在 1 次点击内触达。

---

## 6. 开发与风险管理 (Risk Management)

### 风险 1：单人开发精力不足
*   **对策**：严格执行 MVP。不要自己写 UI 组件库，使用现成的库如 `React Native Paper` 或 `Tamagui`。不要自己造轮子（如单词数据），使用开源数据集。

### 风险 2：服务器成本与性能
*   **对策**：
    *   Supabase 免费层级非常良心，足够支撑 5000+ 活跃用户。
    *   **冷热数据分离**：一年前的学习记录归档，不参与日常排行计算。
    *   **不做文件存储**：严禁用户上传大图或音频，只存文本数据。

### 风险 3：应用商店审核
*   **对策**：Android 直接分发 APK。iOS 需要通过 App Store，如果涉及“虚拟社区”，审核会变严，初期版本可以先隐藏“社区”功能再提审。
